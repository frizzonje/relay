# Arch Linux packaging

Arch has no `.deb`/`.rpm`, so the desktop client is shipped the idiomatic way:
**PKGBUILDs published to the [AUR](https://aur.archlinux.org/)**. These are the
source of truth, versioned with the code; on each release they are copied to the
matching AUR git repos.

Two packages, the standard AUR pattern:

| Package                                        | What it does                                                                 | Needs Rust? |
| ---------------------------------------------- | ---------------------------------------------------------------------------- | ----------- |
| [`relay-desktop-bin`](relay-desktop-bin/PKGBUILD) | Repackages the official `relay_<ver>_amd64.deb` release asset. Seconds to install. | no          |
| [`relay-desktop`](relay-desktop/PKGBUILD)         | Builds the Tauri shell from the tagged sources with the system toolchain.     | yes         |

Both declare `provides=relay-desktop` and conflict with each other, so a user
installs exactly one. Runtime deps are the usual WebKitGTK stack:
`webkit2gtk-4.1 gtk3 libayatana-appindicator`.

> The `-bin` binary is built on the CI's Ubuntu 22.04 runner (glibc 2.35). Arch
> ships a newer glibc, which is backward compatible, so the prebuilt binary runs
> fine — this is the normal trade-off for every `*-bin` AUR package.

## Why no native `.pkg.tar.zst` from CI?

tauri-bundler emits `.deb`/`.rpm`/AppImage but not pacman packages, and Arch
users expect to `paru -S relay-desktop` rather than download a tarball from a
GitHub release. The AUR is the native distribution channel, so that's where the
package lives. The `-bin` PKGBUILD reuses the `.deb` the release already ships —
no extra CI build needed.

## Install (end user)

```bash
paru -S relay-desktop-bin      # prebuilt (fast); or `yay -S …`
paru -S relay-desktop          # build from source
```

## Test a PKGBUILD locally

```bash
cd relay-desktop-bin           # or relay-desktop
makepkg -si                    # build + install
namcap PKGBUILD                # lint the recipe
namcap relay-desktop-bin-*.pkg.tar.zst   # lint the built package
```

`makepkg` refuses to run as root — use a normal user. CI lints both recipes on
every push (see [`desktop.yml`](../../../../.github/workflows/desktop.yml), job
`arch`).

## Cut a release

After a `desktop-v<ver>` release is published (installers uploaded), on an Arch
box:

```bash
./release.sh                   # version from src-tauri/tauri.conf.json
# or: ./release.sh 0.3.8
```

This bumps `pkgver` in both PKGBUILDs, pins the `-bin` checksum with
`updpkgsums` (downloads the `.deb` and records its real hash — the committed
template keeps `SKIP` because the file doesn't exist until the release is out),
and regenerates each `.SRCINFO`. Needs `base-devel` + `pacman-contrib`.

## Publishing to the AUR

`.SRCINFO` is generated by `release.sh`; it is **not** committed here (it only
belongs in the AUR repo, where it must exactly match the PKGBUILD). The AUR repos
are separate from this monorepo:

```bash
git clone ssh://aur@aur.archlinux.org/relay-desktop-bin.git
cp relay-desktop-bin/{PKGBUILD,.SRCINFO} relay-desktop-bin.git/
cd relay-desktop-bin.git && git commit -am "upgpkg: <ver>-1" && git push
# repeat for relay-desktop
```

The first push of a new package name also creates it on the AUR. Set your own
`# Maintainer:` line before publishing.
