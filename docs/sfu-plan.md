# Phase 2 — SFU рядом с p2p (видео на 7 человек «как в Discord»)

> Статус: шаги **A, B, C, D, E сделаны**, дальше — **F** (образ в GHCR и вопрос
> в `install.sh`) и живой прогон. Phase 1 (укрепление
> mesh) сделана — см. `apps/web/lib/voice.ts`. Здесь — про второй транспорт:
> mediasoup-SFU, который живёт **рядом** с mesh, а не вместо него.

## Почему mesh не вывозит 7 с видео

Связь **mesh**: каждый шлёт свой поток каждому напрямую. На N участниках каждый
клиент держит N−1 соединений и аплоадит своё видео N−1 раз.

| N | исходящих видео | аплинк @2.5 Мбит/с | декодеров |
|---|----------------|--------------------|-----------|
| 2 | 1              | 2.5 Мбит/с         | 1         |
| 4 | 3              | 7.5 Мбит/с         | 3         |
| 7 | 6              | **15 Мбит/с**      | 6         |

15 Мбит/с вверх стабильно не тянет ни домашний, ни тем более мобильный аплинк.
Голос на 7 mesh держит (6×~64–128 кбит/с), а видео — нет. Это потолок самой
схемы, кодом не лечится.

**SFU** — медиасервер: каждый шлёт свой поток на сервер **один раз**, сервер
раздаёт остальным. Аплинк участника = константа, сколько бы ни было людей.

## От p2p не отказываемся

Mesh остаётся полноправным транспортом, а не легаси:

- на 2–3 человек он **лучше** SFU: ниже задержка (нет лишнего хопа), ноль
  нагрузки на сервер, медиа не проходит через чужую машину;
- self-host без SFU-профиля обязан работать полностью — это условие
  «поставил одной командой и звони»;
- SFU может лежать/быть выключен, а звонок должен собраться.

## Принятые решения

| Вопрос | Решение |
|---|---|
| Движок | **mediasoup** (библиотека, сигналинг пишем свой поверх socket.io) |
| Кто выбирает режим | **владелец канала**: у голосового канала поле `mode: p2p \| sfu`, по умолчанию `p2p` |
| Размещение | **отдельный сервис `apps/sfu`** в compose, свой контейнер, сокет на пути `/sfu/` |
| Стек сервиса | **NestJS**, как `apps/api` — те же гейтвеи, конфиг и паттерны |
| Self-host | **опциональный профиль** `--profile sfu`, как coturn |
| SFU недоступен | **автофолбэк на p2p** + предупреждение в UI |
| iOS | **заморожен**, в этой фазе не трогаем |

Почему mediasoup, а не готовый сервер: у нас уже есть свой socket.io с
комнатами, авторизацией, presence, гостевыми токенами и ghost-eviction —
сигналинг SFU ложится на этот же слой, а не приносит второй параллельный.
Цена — сигналинг пишем руками; это осознанный размен на контроль.

## Что уже готово в коде (и это хорошая новость)

`apps/web/lib/voice.ts` (2299 строк) наружу отдаёт **только** реактивную
витрину `useVoiceStore` — плитки, состояния кнопок, presence. UI (`VideoGrid`,
`Controls`, `Members`) про `RTCPeerConnection` не знает ничего. Значит второй
транспорт вставляется под этот же стор, без правок UI.

Так же переживают переезд без изменений:

- микшер громкости 0–300% с памятью по тегу — SFU не микширует звук, каждый
  собеседник приходит отдельной дорожкой, Web Audio-цепочка ложится поверх как есть;
- гейт микрофона на `GainNode`, PTT, VAD-обводка «говорит» — они работают
  **до** транспорта, на локальной дорожке;
- захват камеры/экрана, нативный screen-audio на Windows, режимы качества;
- чат, реакции, presence, гостевые ссылки — они к WebRTC не привязаны вообще.

Транспорт-специфичного в `voice.ts` примерно 800–900 строк: `peers` Map,
`createPeer`, обработчики `offer/answer/ice-candidate`, `sendVideoTo`/
`sendScreenTo`, `tuneVideoSender`/`tuneAudioSenders`, `recoverPeer`/
`escalateToRelay`, `drainCandidates`, `updatePeerQuality`.

## Шаг A — выделить `VoiceTransport` — **сделано**

Рефакторинг с неизменным поведением: mesh стал первой реализацией интерфейса.

- `lib/voice/types.ts` — `VoiceTransport` (что транспорт умеет) и `TransportHost`

- `lib/voice.ts` (2299 → ~1600) — дирижёр: устройства, гейт, PTT, захват

- `lib/voice/sfu.ts` — появится на шаге D.

Транспорт не знает про Web Audio и устройства, дирижёр — про `pc`. UI не
тронут вообще. `peer-manager.test.ts` гоняет mesh через публичный API дирижёра
и прошёл без правок — это и есть доказательство, что поведение то же.

## Шаг B — режим в модели канала — **сделано**

- `Channel` в `signaling.gateway.ts` и `packages/shared` получил
  `mode?: 'p2p' | 'sfu'`. Отсутствие поля = `p2p`: старые `registry.json`
  читаются без миграции, и p2p в реестр не пишется вовсе;
- `channel-create` принимает режим, событие `channel-mode` его меняет — права
  те же, что у `channel-delete`: только `removable` и только `type: 'voice'`.
  Дефолтные каналы остаются на p2p — они обязаны работать без медиасервера;
- `GET /api/config` отдаёт `sfu: { available }` (признак — заданный `SFU_URL`).
  `lib/config.ts` теперь кэширует весь ответ, а не только ICE, и отдаёт его
  через `isSfuAvailable()` / хук `useSfuAvailable`;
- в UI: секция «Связь» в модалке создания голосового канала (недоступный режим
  показан выключенным с подписью «Медиасервер не запущен») и пилюля `P2P`/`SFU`
  у канала в сайдбаре — она же переключатель.

Обе ветки пока ведут в mesh: выбор транспорта по `mode` — шаг D.

## Шаг C — сервис `apps/sfu` — **сделано**

Отдельный NestJS-контейнер (как `apps/api`) с mediasoup-воркерами (по числу
ядер, роутер на комнату).

- `media/workers.service.ts` — пул воркеров; смерть воркера = `process.exit(1)`,
  чинить изнутри нечего, пусть перезапустит compose;
- `media/rooms.service.ts` — роутер на комнату, участники, транспорты,
  producer'ы; пустая комната закрывается сразу, повторный вход с тем же id
  закрывает прошлую сессию (иначе после перезагрузки вкладки висит немой дубль);
- `media/media.config.ts` — Opus (stereo+FEC), VP8/VP9/H264, диапазон
  RTC-портов, `announcedAddress` (тот же `TURN_EXTERNAL_IP`, что и у coturn);
- `gateway/sfu.gateway.ts` — сигналинг; `health.controller.ts` — живость.

**Авторизация вышла проще, чем планировалась.** Вместо `GET /api/sfu-token?room=`
пропуск выдаётся событием `sfu-token` **по уже открытому сокету** api: сигналинг
и так знает, кто ты (пропуск или гостевая ссылка), в какой ты комнате и какой у
тебя `socket.id`. Значит `room` и `peerId` берутся из состояния сокета, а не из
запроса — подделать чужой `peerId` или напроситься в чужую комнату нельзя в
принципе, и гости работают даром. HTTP-эндпоинт этого не умел: гость по инвайту
живёт только в сокете, а через express-гейт с `SITE_PASSWORD` не прошёл бы.
Токен живёт 60 секунд, подпись — HMAC на `SFU_SECRET`
(`apps/api/src/sfu/sfu-token.ts` ↔ `apps/sfu/src/token.ts`, формат
байт-в-байт). Пустой `SFU_SECRET` на стороне sfu = не пускать никого.

Пропуск выдаётся только каналам с `mode: 'sfu'` — режим канала из шага B здесь
и начинает работать.

Сокет медиасервера живёт на **пути** `/sfu/`, а не на namespace: в проде оба
сервиса за одним Caddy на одном origin, и различать их надо тем, что видит
прокси. Протокол (ack-стиль socket.io): при подключении сервер сам шлёт
`welcome` (capabilities роутера + кто уже в комнате с их дорожками), дальше
`create-transport` → `connect-transport` → `produce` / `consume` / `resume` /
`close-producer` / `preferred-layers`, обратно — `peer-joined` / `new-producer`
/ `producer-closed` / `peer-left`. `consume` всегда отдаёт consumer на паузе:
клиент цепляет трек и только потом шлёт `resume`.

Инфра для запуска (остальное из F — install.sh и GHCR-образ — впереди): сервис
`sfu` в обоих compose под `profiles: ["sfu"]`, диапазон RTC-портов наружу,
маршрут `/sfu/*` в Caddyfile, `SFU_SECRET`/`SFU_ANNOUNCED_IP` в `.env.example`.
Образ на **bookworm**, а не alpine: mediasoup качает готовый бинарник воркера
под glibc, на musl он бы собирался из исходников.

Проверено (`docker`, без браузера): mediasoup-воркер стартует, транспорт отдаёт
живые ICE-кандидаты (udp+tcp) и DTLS-отпечатки, комнаты изолированы,
`peer-joined`/`peer-left` доходят, мусор в запросах даёт отказ, а не падение;
чужая и протухшая подписи не пускают; **выданный настоящим api токен принимает
настоящий sfu**, и `peerId` в нём совпадает с `socket.id` сигналинга.

Медиа-настройки на будущее: VP8 с **3-слойным simulcast** для камеры (шаг D —
клиентская сторона), демонстрация экрана — отдельный producer с
`contentHint: 'detail'` и жирным битрейтом, без деградации в мыло.

## Шаг D — клиент `lib/voice/sfu.ts` — **сделано**

`mediasoup-client`: `device.load(rtpCapabilities)`, send/recv-транспорты,
`produce` своих дорожек, `consume` чужих. Входящий трек уходит в тот же
`addTile` / `attachRemoteAudio`, что и в mesh — плитки и микшер не замечают
разницы. Своё видео — три слоя simulcast (камера) или один жирный (экран);
роль каждой дорожки едет в `appData.source`, а не угадывается по порядку.

**Выбор транспорта спрашиваем у сервера, а не у своего реестра каналов.** При
входе дирижёр просит пропуск (`sfu-token` с комнатой) и по ответу решает: дали —
SFU, отказали (`not-sfu`/`unavailable`/молчание) — mesh. Причина: гость по
инвайту реестра каналов не получает вовсе (сервер шлёт его только своим), и,
решай мы локально, гость ушёл бы в mesh, пока остальные в SFU, — то есть остался
бы без звука. Поэтому же `sfu-token` теперь принимает комнату в запросе: узнать
транспорт надо ДО `join`, иначе ответный `peers` придёт в никуда. Секрета в
комнате нет — войти в любой голосовой канал клиент и так вправе, а `peerId`
по-прежнему берётся из сокета.

Мелочь, но важная для микшера: `attachRemoteAudio` третьим аргументом принимает
не `mid`, а **слот** — ключ порядка дорожек внутри собеседника. Mesh кладёт туда
`mid` (роль иначе не узнать), SFU — фиксированный номер по `source`. Дирижёр
не изменился.

**Adaptive subscription** — плитка на весь экран просит верхний слой, остальные
нижний (`preferred-layers`). Шов получил необязательный `focusChanged(id)`:
дирижёр дёргает его на смене фокуса, mesh не реализует.

Что осталось на E (и почему в D этого нет): `pollStats` у SFU **пуст** —
палочки и пинг в mesh меряют связь до собеседника, которой здесь нет, а
показывать старые цифры хуже, чем не показывать; `retuneVideo` (тумблер
качество/ФПС) в SFU упирается в слои, а не в `degradationPreference`; лестница
восстановления при `connectionstatechange === 'failed'` пока только пишет
статус.

## Шаг E — то, что ломается при смене транспорта — **сделано**

**Палочки качества.** Семантика поменялась честно, а не подменилась: RTT и
потери в SFU-режиме — это **наш канал до сервера**, половину пути собеседника мы
не видим в принципе. Поэтому в тултипе «Соединение: через сервер», строка
исходящего битрейта «к нему» исчезает (своё медиа уходит один раз, общее на
всех), зато появляется **слой** — и не наша заявка в `preferred-layers`, а факт:
сервер шлёт `layerschange`, мы его пересылаем клиенту. Пороги, проценты потерь
и кбит/с вынесены в `lib/voice/quality.ts` — общие для обоих транспортов, иначе
одни и те же четыре палочки значили бы в двух режимах разное.

**Лестница восстановления.** В mesh она чинила связь с каждым собеседником
отдельно; здесь собеседник один — сервер, — поэтому лестница одна на звонок:
`restart-ice` (новый обработчик на sfu) → пересборка обоих транспортов поверх
живого сокета (`close-transport`, иначе старые висят до дисконнекта, а остальные
слушают мёртвые дорожки) → сдаться. Ступени по 8 с, `disconnected` получает
фору в 4 с (часто проходит сам). Отдельно сторож входа: не поднялись за 12 с —
это отказ, а не «ещё чуть-чуть».

**Автофолбэк — и его граница.** Сдаться транспорт может, а решать, куда ехать,
не вправе: состав канала и комнату знает только дирижёр. Поэтому шов получил
`transportLost('setup' | 'lost')`, а политика живёт в `voice.ts`:

- **на входе** (`setup`) — всегда в p2p, тост «Медиасервер недоступен — звоним
  напрямую»: человек ещё никого не слышал, ждать ему нечего;
- **в звонке** (`lost`) — в p2p только при **≤3 собеседниках**. Дальше mesh не
  спасение, а ровно та боль, ради которой затевался SFU: вместо этого канал
  ждёт сервер и раз в 5 с пробует получить пропуск заново.

**Смена режима на живом канале.** api шлёт `voice-mode` **в комнату** (а не
только с реестром каналов) — ради гостей: реестра у них нет, а разъехаться в
транспортах нельзя, это тихая потеря звука. Дальше — общий переезд `remigrate()`:
leave старого транспорта → плитки собеседников снимаются → новый пропуск → join.
Через него же идут фолбэк, возвращение сервера и реконнект сокета с новым id
(старый пропуск умирает вместе с прежним `socket.id`).

Проверено: 4 теста в `lib/voice-fallback.test.ts` гоняют политику через
публичный API дирижёра — заглушка SFU говорит «не вывез», а тест смотрит, куда
уехал звонок: в mesh на входе, в mesh при малом составе, **не** в mesh при
большом, и что круги ожидания не срываются в mesh, пока сервер лежит.

## Чего это стоит

Серверу 7 человек — копейки по CPU (SFU не транскодит, только форвардит).
Главный расход — аплинк: ~7 входящих потоков и раздача каждому; при simulcast
и адекватных битрейтах укладывается в обычный VPS-канал. По усилиям mediasoup
дороже готового сервера примерно на шаг C — свой сигналинг вместо чужого SDK.
