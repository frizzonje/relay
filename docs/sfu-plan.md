# Phase 2 — SFU рядом с p2p (видео на 7 человек «как в Discord»)

> Статус: **план, не реализовано.** Phase 1 (укрепление mesh) сделана — см.
> `apps/web/lib/voice.ts`. Здесь — про второй транспорт: mediasoup-SFU, который
> живёт **рядом** с mesh, а не вместо него.

## Почему mesh не вывозит 7 с видео

Связь **mesh**: каждый шлёт свой поток каждому напрямую. На N участниках каждый
клиент держит N−1 соединений и аплоадит своё видео N−1 раз.

| N | исходящих видео | аплинк @2.5 Мбит/с | декодеров |
|---|----------------|--------------------|-----------|
| 2 | 1              | 2.5 Мбит/с         | 1         |
| 4 | 3              | 7.5 Мбит/с         | 3         |
| 7 | 6              | **15 Мбит/с**      | 6         |

15 Мбит/с вверх стабильно не тянет ни домашний, ни тем более мобильный аплинк.
Голос на 7 mesh держит (6×~64–128 кбит/с), а видео — нет. Это потолок самой
схемы, кодом не лечится.

**SFU** — медиасервер: каждый шлёт свой поток на сервер **один раз**, сервер
раздаёт остальным. Аплинк участника = константа, сколько бы ни было людей.

## От p2p не отказываемся

Mesh остаётся полноправным транспортом, а не легаси:

- на 2–3 человек он **лучше** SFU: ниже задержка (нет лишнего хопа), ноль
  нагрузки на сервер, медиа не проходит через чужую машину;
- self-host без SFU-профиля обязан работать полностью — это условие
  «поставил одной командой и звони»;
- SFU может лежать/быть выключен, а звонок должен собраться.

## Принятые решения

| Вопрос | Решение |
|---|---|
| Движок | **mediasoup** (библиотека, сигналинг пишем свой поверх socket.io) |
| Кто выбирает режим | **владелец канала**: у голосового канала поле `mode: p2p \| sfu`, по умолчанию `p2p` |
| Размещение | **отдельный сервис `apps/sfu`** в compose, свой контейнер и namespace `/sfu` |
| Стек сервиса | **NestJS**, как `apps/api` — те же гейтвеи, конфиг и паттерны |
| Self-host | **опциональный профиль** `--profile sfu`, как coturn |
| SFU недоступен | **автофолбэк на p2p** + предупреждение в UI |
| iOS | **заморожен**, в этой фазе не трогаем |

Почему mediasoup, а не готовый сервер: у нас уже есть свой socket.io с
комнатами, авторизацией, presence, гостевыми токенами и ghost-eviction —
сигналинг SFU ложится на этот же слой, а не приносит второй параллельный.
Цена — сигналинг пишем руками; это осознанный размен на контроль.

## Что уже готово в коде (и это хорошая новость)

`apps/web/lib/voice.ts` (2299 строк) наружу отдаёт **только** реактивную
витрину `useVoiceStore` — плитки, состояния кнопок, presence. UI (`VideoGrid`,
`Controls`, `Members`) про `RTCPeerConnection` не знает ничего. Значит второй
транспорт вставляется под этот же стор, без правок UI.

Так же переживают переезд без изменений:

- микшер громкости 0–300% с памятью по тегу — SFU не микширует звук, каждый
  собеседник приходит отдельной дорожкой, Web Audio-цепочка ложится поверх как есть;
- гейт микрофона на `GainNode`, PTT, VAD-обводка «говорит» — они работают
  **до** транспорта, на локальной дорожке;
- захват камеры/экрана, нативный screen-audio на Windows, режимы качества;
- чат, реакции, presence, гостевые ссылки — они к WebRTC не привязаны вообще.

Транспорт-специфичного в `voice.ts` примерно 800–900 строк: `peers` Map,
`createPeer`, обработчики `offer/answer/ice-candidate`, `sendVideoTo`/
`sendScreenTo`, `tuneVideoSender`/`tuneAudioSenders`, `recoverPeer`/
`escalateToRelay`, `drainCandidates`, `updatePeerQuality`.

## Шаг A — выделить `VoiceTransport` — **сделано**

Рефакторинг с неизменным поведением: mesh стал первой реализацией интерфейса.

- `lib/voice/types.ts` — `VoiceTransport` (что транспорт умеет) и `TransportHost`
  (что он вправе спросить у дирижёра). Двусторонний шов: наружу — публикация
  локальных дорожек, внутрь — плитки, микшер, статусы, метрики;
- `lib/voice/mesh.ts` (~700 строк) — весь `RTCPeerConnection`: perfect
  negotiation, лестница восстановления, палочки качества, тюнинг битрейтов;
- `lib/voice.ts` (2299 → ~1600) — дирижёр: устройства, гейт, PTT, захват
  камеры/экрана, микшер входящего звука, VAD, плитки;
- `lib/voice/sfu.ts` — появится на шаге D.

Транспорт не знает про Web Audio и устройства, дирижёр — про `pc`. UI не
тронут вообще. `peer-manager.test.ts` гоняет mesh через публичный API дирижёра
и прошёл без правок — это и есть доказательство, что поведение то же.

## Шаг B — режим в модели канала

- `Channel` в `signaling.gateway.ts` и `packages/shared` получает
  `mode?: 'p2p' | 'sfu'` (отсутствует = `p2p`, старые `registry.json` читаются
  без миграции);
- `channel-create` принимает режим, новое событие `channel-mode` его меняет —
  права те же, что у `channel-delete` (владелец/`removable`);
- `GET /api/config` отдаёт `sfu: { available: boolean }` — фронт заранее знает,
  поднят ли сервис, и рисует бейдж режима на канале;
- в UI: селектор режима в создании/настройках голосового канала + подпись
  «через сервер» / «напрямую» рядом с каналом.

## Шаг C — сервис `apps/sfu`

Отдельный NestJS-контейнер (как `apps/api`) с mediasoup-воркерами (по числу
ядер, роутер на комнату), socket.io namespace `/sfu`.

Авторизация: клиент берёт у api короткоживущий JWT (`GET /api/sfu-token?room=`,
подпись общим `SFU_SECRET` из env, рядом с `TURN_CREDENTIAL`), sfu его
проверяет. Пароль сайта, закрытые серверы и гостевые токены остаются целиком
на стороне api — sfu ничего про них не знает, только проверяет подпись и
`room` в клейме.

Протокол (стандартный mediasoup-хендшейк, ack-стиль socket.io):
`rtp-capabilities` → `create-transport` (send/recv) → `connect-transport` →
`produce` / `consume` / `resume` + серверные `new-producer` / `producer-closed`.

Медиа-настройки: Opus для звука; VP8 с **3-слойным simulcast** для камеры;
демонстрация экрана — отдельный producer с `contentHint: 'detail'` и жирным
битрейтом, без деградации в мыло. H264 держим в capabilities роутера как
страховку для чужих браузеров.

## Шаг D — клиент `lib/voice/sfu.ts`

`mediasoup-client`: `device.load(rtpCapabilities)`, send/recv-транспорты,
`produce` своих дорожек, `consume` чужих. Входящий трек уходит в тот же
`addTile` / `attachRemoteAudio`, что и в mesh — плитки и микшер не замечают
разницы.

Отдельно: **adaptive subscription** — плитка на весь экран просит верхний слой,
мелкая плитка нижний (`consumer.setPreferredLayers`). Это то, ради чего
simulcast и затевался.

## Шаг E — то, что ломается при смене транспорта

- **Палочки качества.** Сейчас `net` — это RTT/потери **до собеседника**. В SFU
  их нет: есть качество до сервера + состояние слоя. Семантику меняем честно:
  палочки = свой канал до сервера, в тултипе — слой и битрейт входящего.
- **Восстановление связи.** `recoverPeer`/`escalateToRelay` (ICE-restart →
  relay-only) в SFU заменяются на `transport.restartIce()` + пересоздание
  транспорта. Лестница проще, но её надо написать заново, а не «оно само».
- **Автофолбэк.** Не собрался транспорт за таймаут или sfu недоступен → тост
  «Медиасервер недоступен, звоним напрямую» и `join` через mesh. Логика живёт
  в дирижёре `voice.ts`, а не внутри реализаций.
- **Смена режима на живом канале.** Владелец переключил `mode` при активном
  звонке → всем участникам мягкий переезд (leave старого транспорта → join
  нового), звук пропадёт на пару секунд. Проще, чем поддерживать мост.

## Шаг F — инфра

- сервис `sfu` в `docker-compose.yml` под `profiles: ["sfu"]`, образ в
  `release-images` workflow и GHCR;
- порты: UDP+TCP диапазон RTC (напр. 40000–40100) на хосте; `announcedIp` =
  `TURN_EXTERNAL_IP` или `SERVER_HOST` — на облачных VM за 1:1 NAT без него
  медиа не пойдёт (те же грабли, что у coturn);
- Caddy: `/sfu/*` → `sfu:3100` (WS);
- `install.sh`: вопрос «включить медиасервер (нужен для звонков 4+)?» и
  подсказка про порты в фаерволе;
- **coturn SFU не переиспользует** — mediasoup сам себе ICE-сервер и не умеет
  ходить клиентом через внешний TURN. Для сетей, где наружу пускают только
  TCP/443, у SFU-режима остаётся ICE-TCP на нашем порту; если и он закрыт —
  сработает фолбэк на p2p, где TURN-over-TLS через coturn пройдёт. Ещё один
  аргумент не выпиливать mesh.

## Чего это стоит

Серверу 7 человек — копейки по CPU (SFU не транскодит, только форвардит).
Главный расход — аплинк: ~7 входящих потоков и раздача каждому; при simulcast
и адекватных битрейтах укладывается в обычный VPS-канал. По усилиям mediasoup
дороже готового сервера примерно на шаг C — свой сигналинг вместо чужого SDK.
